## ADDED Requirements

### Requirement: Deterministic tooling owns pixel comparison

Pixel-level screenshot comparison SHALL be performed by deterministic visual regression tooling (Chromatic for Storybook-first component flows; Playwright visual comparisons for repository-local page or flow assertions). LLM-based reasoning SHALL NOT be the source of truth for visual diffs.

#### Scenario: Diff is produced by tooling
- **WHEN** any artifact undergoes the Visual Regression Gate
- **THEN** a baseline image, an actual image, and a diff artifact exist, generated by Chromatic or Playwright
- **AND** the gate's pass/fail signal derives from the tool's diff output, not from a model's interpretation

#### Scenario: Tool selection is recorded
- **WHEN** an artifact has a visual regression configured
- **THEN** its repository-local config (Storybook + Chromatic, or a Playwright spec) declares which tool produced its baselines

### Requirement: Visual reviewer interprets, does not measure

`visual-reviewer` SHALL read the diff artifacts produced by the tooling, classify the likely cause (token drift, spacing drift, state miscoverage, font fallback, anti-aliasing noise, intentional change), and translate that classification into a recommendation for `design-system-guardian`. It SHALL NOT produce its own pixel measurements or override the tool's diff signal.

#### Scenario: Reviewer classifies a real diff
- **WHEN** Chromatic reports a diff above tolerance for a primitive
- **THEN** `visual-reviewer` reads the baseline, actual, and diff images and emits a classification with rationale

#### Scenario: Reviewer cannot bypass tooling
- **WHEN** `visual-reviewer` produces a recommendation
- **THEN** the recommendation cites the tool's diff output and does NOT claim a pass that contradicts a tool failure

### Requirement: Baselines are stored and reviewable

For each primitive, set, and screen with visual regression, baseline images SHALL be stored in a location reviewable in version control (Chromatic's hosted baselines linked via `chromatic.config` and PR comments; Playwright snapshots committed under `ui-lab/snapshots/`). Baseline updates SHALL require explicit approval recorded in the changelog or PR.

#### Scenario: Baseline is captured on first promotion
- **WHEN** a primitive passes the Guardian Approval Gate for the first time
- **THEN** a baseline image set is captured and stored

#### Scenario: Baseline update requires approval
- **WHEN** a diff exceeds tolerance and is judged to be an intentional change
- **THEN** the baseline update MUST be explicitly approved by `design-system-guardian`
- **AND** the approval is recorded with a reference to the rule, token, or component change that motivated it

### Requirement: Visual Regression Gate ties to canonical change

When a visual diff exceeds tolerance, the gate SHALL pass only if the diff is traceable to an approved change in `canonical/rule-engine/changelog.md` or `migrations/`. Unexplained diffs SHALL fail the gate.

#### Scenario: Diff matches a canonical change
- **WHEN** a token value changes per a recorded rule or migration
- **AND** the resulting diff matches the expected scope
- **THEN** the gate passes after baseline update

#### Scenario: Unexplained diff fails the gate
- **WHEN** a diff appears with no corresponding canonical change
- **THEN** the gate fails and the artifact is rolled back or the source of drift is investigated

### Requirement: Tolerance is configured per component class

Each component class SHALL have a configured pixel-difference tolerance to avoid false positives from anti-aliasing or font-rendering noise. Tolerance values SHALL live in repository config (e.g., Chromatic project config, Playwright `toMatchSnapshot` options) and SHALL be reviewable.

#### Scenario: Default tolerance is set
- **WHEN** a new component class is introduced
- **THEN** a tolerance value is recorded in repository config
- **AND** the value is referenced from the component's manifest if non-default

#### Scenario: Tolerance change requires approval
- **WHEN** a tolerance value is widened
- **THEN** the change is recorded with rationale, since looser tolerance reduces gate signal
