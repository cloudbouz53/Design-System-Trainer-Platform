# Use Semantic Color Tokens

**Rule ID:** TOK-001
**Status:** active
**Version:** 1.0.0
**Source:** inferred-rules.md §Use semantic color tokens; materials-notes.md §Color; raw-observations.md §Color — human approval by guardian 2026-05-01
**Scope:** `ui-lab/**`
**Priority:** critical

## Do

Reference only canonical semantic token names (CSS custom properties from `design-system/canonical/tokens/colors.md`) for all color values in component and app implementations.

```css
/* Good */
background-color: var(--color-background);
color: var(--color-text-primary);
border-color: var(--color-border);
```

## Do Not

Use raw hex colors, RGB/HSL/OKLCH literals, or named CSS colors (`red`, `blue`, `white`) in component or app styles — except where explicitly noted as acceptable exceptions in the component's manifest.

```css
/* Bad */
background-color: #ffffff;
color: #0a0a0a;
border-color: rgb(230, 230, 230);
```

## Why

Token-based color enables theming, dark mode, and design evolution without modifying component implementations. Hardcoded values create hidden dependencies on specific color values that break when the design system evolves.

## Validation

Grep the implementation files for `#[0-9a-fA-F]{3,8}`, `rgb(`, `rgba(`, `hsl(`, `oklch(`, and CSS named colors not inside comments. Any match that resolves to a color value outside the token system fails this rule. Acceptable exception: `transparent`, `currentColor`, `inherit`.

## Impact

Violations affect all components and apps in `ui-lab/`. A widespread hardcoded-color failure would require re-reviewing every artifact in the dependency graph.

## Good Example

```css
.button-primary {
  background-color: var(--color-interactive);
  color: white; /* acceptable: white on interactive is documented exception */
  border: none;
}
```

## Bad Example

```css
.button-primary {
  background-color: #0070f3; /* hardcoded — fails TOK-001 */
  color: #ffffff;            /* hardcoded — fails TOK-001 */
}
```
