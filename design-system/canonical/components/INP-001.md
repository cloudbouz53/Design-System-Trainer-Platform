# Input Must Have Accessible Label

**Rule ID:** INP-001
**Status:** active
**Version:** 1.0.0
**Source:** inferred-rules.md §Input must have accessible label; component-notes.md §Input §Accessibility; raw-observations.md §Input — human approval by guardian 2026-05-01
**Scope:** `ui-lab/primitives/input/**`
**Priority:** critical

## Do

Always associate an input with either a visible `<label>` (via `htmlFor`/`id`) or an `aria-label` attribute. Do both when possible.

```tsx
<label htmlFor="email-input">Email address</label>
<input id="email-input" type="email" />
```

## Do Not

Use placeholder text as the sole label. Placeholders disappear on input and are not reliably announced by all screen readers.

```tsx
/* Bad — placeholder only */
<input type="email" placeholder="Enter your email" />
```

## Why

Screen readers require explicit label association to announce the input's purpose. Without a label, users relying on assistive technology cannot determine what an input expects.

## Validation

Check that every `<input>` element in the implementation has either:
1. An `id` attribute paired with a `<label htmlFor>` element in the same component, OR
2. An `aria-label` attribute, OR
3. An `aria-labelledby` attribute pointing to an existing element

If only a `placeholder` is present without one of the above, fail this rule.

## Impact

Affects `ui-lab/primitives/input/` and any component set or app that uses Input. Failure immediately fails the Accessibility Gate.

## Good Example

```tsx
function Input({ id, label, ...props }: InputProps) {
  return (
    <div>
      <label htmlFor={id}>{label}</label>
      <input id={id} {...props} />
    </div>
  );
}
```

## Bad Example

```tsx
function Input({ placeholder, ...props }: InputProps) {
  return <input placeholder={placeholder} {...props} />;
  // No label — fails INP-001
}
```
